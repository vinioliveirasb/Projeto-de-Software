# 📚 Projeto de Software — Aula 2

## Modelagem de Sistemas de Software

Para a construção de **sistemas de software complexos**, é necessário criar **modelos antes do início do projeto**.

### Motivos para utilizar modelos

- **Limitação humana para lidar com complexidade**  
  Sistemas grandes são difíceis de compreender completamente.  
  Por isso podem existir **vários modelos**, cada um representando uma **perspectiva do sistema**.

- **Comunicação entre os envolvidos**  
  O desenvolvimento de software envolve muitas atividades e pessoas.  
  Grande parte das informações sobre o sistema é representada por **modelos**.

- **Redução de custos**  
  Erros são inevitáveis no desenvolvimento (individuais ou de comunicação).  
  Modelos ajudam a identificar problemas **antes da implementação**, reduzindo custos.

- **Previsão do comportamento do sistema**  
  Os modelos funcionam como um **“laboratório”**, permitindo testar diferentes soluções antes da construção do sistema.

# Paradigma da Orientação a Objetos

Por meio de uma **analogia biológica**, **Alan Kay** imaginou um sistema de software que funcionasse como um **organismo vivo**.

Nesse sistema:

- Cada **“célula”** seria um objeto.
- As células **interagem enviando mensagens** entre si.
- Cada célula funciona como uma **unidade autônoma**.
- A cooperação entre os objetos permite que o sistema resolva tarefas.

---

# Princípios da Orientação a Objetos

- Qualquer coisa é um **objeto**
- Objetos realizam tarefas através de **requisições de serviços de outros objetos**
- Cada objeto pertence a uma **classe**
- Uma **classe agrupa objetos similares**
- A classe é um **repositório de comportamentos**
- Classes são organizadas em **hierarquias**

---

# Funcionamento dos Objetos

## Operação

Uma **operação** é uma ação que um objeto sabe realizar quando solicitada.

Sempre existe um **estímulo** que dispara a execução.

---

## Mensagem

Quando um objeto solicita que outro execute uma operação, ele envia uma **mensagem**.

Ou seja:

Objetos se comunicam enviando mensagens para realizar tarefas.

---

## Estado

O **estado de um objeto** é o conjunto de valores de seus atributos em um determinado momento.

Uma mensagem pode **alterar o estado** do objeto.

---

# Abstração

A **abstração** é um processo mental no qual focamos nos **aspectos mais importantes** de algo, ignorando detalhes menos relevantes.

Ela é fundamental na orientação a objetos.

---

# Conceitos da Abstração em OO

## Encapsulamento

- Protege os dados do objeto
- Garante **segurança e privacidade**

---

## Polimorfismo

- Permite que diferentes objetos **respondam de formas diferentes à mesma mensagem**

---

## Generalização (Herança)

- Permite criar **hierarquias de classes**
- Relação **genérico → específico**

---

## Composição

- Um **objeto pode conter outros objetos como atributos**

---

# UML e Visões de um Sistema

## Visão de Casos de Uso

- Mostra o sistema do **ponto de vista externo**
- Representa a interação entre **usuários e sistema**
- Direciona o desenvolvimento das outras visões

---

## Visão de Projeto

Enfatiza características que dão suporte:

- Estrutura do sistema
- Comportamento das funcionalidades

---

## Visão de Implementação

Relacionada ao **gerenciamento de versões** e organização do sistema em:

- Componentes
- Subsistemas

Define também a **distribuição física do sistema**.

---

## Visão de Processo

Enfatiza:

- Concorrência
- Paralelismo
- Sincronização
- Desempenho do sistema

---

# Artefatos de Software

Em um processo de desenvolvimento que utiliza **UML**, são criados diversos documentos chamados:

**Artefatos de Software**

Esses artefatos compõem as **visões do sistema**.

# Etapas do Desenvolvimento de Software

## Comunicação

Identificação e levantamento de **requisitos do sistema**.

---

## Planejamento

Definição de:

- Tempo
- Custo
- Protótipos

---

## Projeto

Modelagem do sistema utilizando **UML**.

---

## Construção

Implementação do **código do sistema**.

---

## Entrega

Disponibilização do sistema para o usuário final.

---

# Domínio do Sistema

O **domínio** corresponde à área de conhecimento relacionada ao sistema.

Exemplo:

- Sistema bancário → **domínio bancário**

O domínio define o que é **relevante para o sistema**.

---

# Tipos de Requisitos

## Requisitos Funcionais

Definem **o que o sistema deve fazer**.

---

## Requisitos Não Funcionais

Definem **como o sistema deve funcionar**, considerando restrições como:

- Segurança
- Portabilidade
- Usabilidade

---

## Requisitos Normativos

São **regras, leis ou normas** que impõem restrições ao sistema.

---

# Fase de Análise

A fase de análise utiliza informações da **comunicação e planejamento**.

Objetivo:

Realizar um **estudo detalhado dos requisitos** e construir modelos do sistema.

Características:

- Foco nos aspectos **lógicos**
- Independente de tecnologia

Principais diagramas:

- **Diagrama de Casos de Uso**
- **Diagrama de Classes**

---

# Fase de Projeto

Define **como o sistema funcionará** para atender aos requisitos.

Aqui são adicionadas as **restrições tecnológicas**, como:

- Arquitetura física do sistema
- Padrões de interface gráfica
- Algoritmos específicos

O projeto possui duas etapas:

---

## Projeto de Arquitetura

Distribui as **classes do sistema** em:

- Subsistemas
- Componentes

---

## Projeto Detalhado

Modela a **colaboração entre objetos** para implementar as funcionalidades.

---

# Fase de Implementação

O sistema é **codificado**, ou seja:

A descrição computacional obtida no projeto é **traduzida para código executável** utilizando linguagens de programação.

---

# Fase de Teste

Realiza atividades para **verificar se o sistema atende às especificações**.

Um dos resultados dessa fase são os **relatórios de testes**, contendo erros encontrados no software.

---

## Teste Unitário

Testa partes individuais do código.

Exemplo:

- Cada método de uma classe possuir **uma classe de teste específica**.

---

## Teste de Integração

Testa a **interação entre componentes do sistema** após os testes unitários.

Em sistemas orientados a objetos, o foco pode ser:

- operações
- interações entre objetos
